# Customer authentication

## Architecture

This project uses the existing Payload `users` collection for both storefront
customers and administrators. This is the shared-collection (Case B) design:
Payload remains authoritative for users, roles, account status, profiles,
password credentials, and sessions. The ecommerce plugin continues to reference
`users`, so carts, addresses, orders, and wishlists do not need data migration.

Public registration does **not** use generic `POST /api/users`. Collection
creation is restricted to administrators; `POST /api/auth/signup` validates an
allow-listed body and forces `roles: ['customer']` and `accountStatus: 'active'`.
Only users with the `admin` role may enter Payload Admin.

The first Payload administrator must match `BOOTSTRAP_ADMIN_EMAIL` when using
the first-user registration screen. Public signup can never bootstrap an admin.

## Email and password

- Signup: `POST /api/auth/signup`
- Login: `POST /api/auth/login`
- Current user: `GET /api/auth/session`
- Verify: `/verify-email?token=...`
- Resend verification: `POST /api/auth/resend-verification`
- Forgot password: `POST /api/auth/forgot-password`
- Reset password: `POST /api/auth/reset-password`

Payload hashes passwords, enforces verification, limits login attempts to five,
locks an account for 15 minutes, and persists seven-day Payload sessions. The
browser receives only an HTTP-only Payload cookie; login tokens are removed
from JSON responses. Existing users are marked verified by the additive
migration so enabling the policy does not lock out pre-existing customers or
administrators.

Forgot-password responses are deliberately generic. A completed reset adds the
password method without removing Google and revokes previous Payload and OAuth
application sessions. This is also the recovery path for a Google-only user.

## Google OpenID Connect

Google uses Authorization Code flow with PKCE, a random state, and a nonce:

1. `GET /api/auth/google` creates a signed, ten-minute HTTP-only flow cookie.
2. Google returns to `GET /api/auth/google/callback`.
3. `google-auth-library` exchanges the code and verifies signature, issuer,
   audience, expiry, nonce, subject, and verified email.
4. `auth-identities` resolves `google:<sub>`. A cryptographically verified email
   may link to an existing customer, but never automatically to an administrator.
5. A 32-byte opaque app token is created. Only its HMAC-SHA-256 digest is stored
   in `oauth-sessions`; the raw token is held in `app_oauth_session`.

No Google access token, ID token, refresh token, authorization code, password,
or raw application session token is persisted in application collections or
browser storage.

Identity/user/session creation shares a Payload request transaction. Unique
indexes on user email, provider key, and token hash prevent duplicate records.
A uniqueness race is retried once after rollback.

## Cookies and logout

Both authentication cookies are HTTP-only, `SameSite=Lax`, scoped to `/`, and
`Secure` in production. `AUTH_COOKIE_DOMAIN` is optional and should stay blank
for a single hostname.

- Payload cookie: generated by Payload, seven days.
- `app_oauth_session`: opaque Google application session, seven days.
- `app_google_oauth_flow`: signed OAuth state/nonce/PKCE data, ten minutes.

`POST /api/auth/logout` revokes the current Payload and OAuth session and clears
all auth flow cookies. `POST /api/auth/logout-all` clears every Payload session
and revokes every active OAuth session for the user.

## Account blocking and access

Set `accountStatus` to `blocked` in Payload Admin. Password login hooks and the
OAuth resolver reject blocked accounts. Existing OAuth sessions are rejected on
lookup. Existing Payload JWT sessions are invalidated when Payload loads the
blocked account. Customers can read only themselves and may update profile
fields only through `PATCH /api/auth/profile`; email, roles, account status,
provider identities, sessions, and authentication metadata are server-managed.

Audit records are stored in `auth-audit-events` and are readable only by admins.
Rate-limit buckets are stored in `auth-rate-limits` so limits survive serverless
instance changes. Identifiers and IP addresses are HMAC-hashed.

## Environment variables

```env
PAYLOAD_SECRET=
DATABASE_URL=
DATABASE_URI=

NEXT_PUBLIC_APP_URL=http://localhost:3000
APP_URL=http://localhost:3000
NEXT_PUBLIC_SERVER_URL=http://localhost:3000
PAYLOAD_PUBLIC_SERVER_URL=http://localhost:3000

NEXT_PUBLIC_GOOGLE_AUTH_ENABLED=false
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
GOOGLE_REDIRECT_URI=http://localhost:3000/api/auth/google/callback

AUTH_SESSION_SECRET=
AUTH_COOKIE_DOMAIN=
```

Use independent high-entropy values for `PAYLOAD_SECRET` and
`AUTH_SESSION_SECRET`. Never prefix secrets with `NEXT_PUBLIC_`. Production
startup rejects a missing session secret and requires PostgreSQL.

## Google Cloud setup

1. Create or select a Google Cloud project.
2. Configure the OAuth consent screen and request only `openid email profile`.
3. Create an OAuth client of type **Web application**.
4. Add `http://localhost:3000` as the development JavaScript origin.
5. Add the exact production HTTPS origin.
6. Add `http://localhost:3000/api/auth/google/callback` as a development
   authorized redirect URI.
7. Add `<production-origin>/api/auth/google/callback` as the production
   authorized redirect URI.
8. Copy the client ID and secret to the deployment environment and set
   `NEXT_PUBLIC_GOOGLE_AUTH_ENABLED=true`.
9. Restart or redeploy after changing variables.

For `redirect_uri_mismatch`, compare the full URI character-for-character,
including scheme, hostname, port, path, and trailing slash. The configured
`GOOGLE_REDIRECT_URI` must be one of the authorized Google redirect URIs.

## Troubleshooting and deployment checklist

- SMTP must be configured for verification and password reset email.
- Production must use HTTPS.
- The frontend and Payload API are same-origin; do not broaden CORS or cookies
  unless the deployment architecture requires it.
- Confirm the Google callback exactly matches Cloud Console.
- Inspect browser cookies to confirm `HttpOnly`, `Secure` in production, and
  absence of tokens in Local Storage, Session Storage, or IndexedDB.
- Verify `/admin` still accepts only administrators.
- Test a blocked user, Google/email linking, logout-all, verification, and
  password reset after every authentication release.
- Revoke OAuth application sessions by setting `revokedAt` or using logout-all.
  Payload sessions may be cleared from the user's `sessions` array.
